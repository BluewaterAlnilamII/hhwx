1\. 架构与技术栈规范

前端框架：React + Next.js (App Router)。所有涉及浏览器交互（如 onClick、LocalStorage 读取）的组件，必须在文件顶部显式声明 "use client"。

样式方案：Tailwind CSS + Shadcn UI。

状态管理：

Zustand：用于管理全局状态（如用户登录态、角色选择界面的进度）。

React Hooks (useReducer 或 useState)：用于管理单局黑白棋的高频状态（如 8x8 棋盘矩阵、当前回合归属）。

数据与鉴权：后端采用 Supabase。对局进度需实时保存在前端 LocalStorage 中（注意：读取 LocalStorage 恢复进度时，必须在 useEffect 中执行，以防止 Next.js 产生 Hydration Error）。

2\. 核心视觉与 UI 布局预留

主色调与背景：网页背景色设为 #FFEE22，并全局叠加一层白色半透明雾化滤镜（可通过 CSS backdrop-filter 或半透明遮罩实现）以防刺眼。背景需实现图案（原版与透明版交替）自上而下缓慢飘落的 CSS 动画。

中心区域：居中放置标准 8x8 黑白棋盘。

角色立绘区（组件化规范）：

左上角（AI 对手）与右下角（玩家）。

占位符要求：在真实美术资产接入前，请使用带有固定宽高比的 div 色块作为占位图。需将其封装为独立的 <CharacterAvatar /> 组件，暴露出 characterId 和 emotion（情绪状态）等 Props，为后续接入真实图片或 Live2D Canvas 预留清晰的 DOM 结构。

立绘旁需附带交互式气泡对话框（UI 层级需高于背景）。

工具栏：右上角提供“登录/注册”入口，登录后自动切换为账号管理菜单。

底部区域：独立的游戏评论区模块。

3\. 游戏核心逻辑与边缘规则 (Edge Cases)

基础规则：实现标准黑白棋的落子、八个方向的合法性校验及翻转规则。

回合跳过 (Pass) 机制：若当前回合方在棋盘上没有任何合法的落子点，系统需自动跳过该回合，将控制权交还给对方，并在 UI 中央短暂弹出提示（如“无处落子，回合跳过”）。

游戏结束与结算：

触发条件：棋盘 64 格全满，或双方均无合法落子点时游戏结束。

结算动作：弹出结算面板，展示双方最终棋子数量和胜负结果。提供「重新开始」和「返回选角」按钮。同时自动清除 LocalStorage 中的当前对局缓存。

交互防抖与状态机锁定：在玩家成功落子后，直到 AI 思考结束并完成落子的整个期间，棋盘必须进入 disabled 状态，屏蔽玩家的任何点击事件，防止误触导致逻辑崩溃。

4\. 角色设定、AI 行为树与性能要求

游戏开始前，玩家需从 5 名角色中选定 1 名出战，再从剩余 4 名中选择 1 名作为 AI 对手。以下为 AI 对手的逻辑规范：

弦巻 こころ (Kokoro)：依靠直觉的战略型选手，热衷于通过对边角的战略性争夺来取得优势。

瀬田 薫 (Kaoru) \[性能高危]：AI 强度最高，思考时间长。采用深度优先搜索（DFS）或极小极大算法（Minimax），以绝对战胜玩家为唯一目标。性能规范：为防止主线程阻塞导致网页假死或背景动画卡顿，此算法必须设置严格的搜索深度上限（如 4-6 步），或封装进 Web Worker 中异步执行。

北沢 はぐみ (Hagumi)：黑白棋初学者，常以“气势”一次性翻转大量对手的棋子，但不擅长争夺边角上的战术。

松原 花音 (Kanon)：行动力战术的践行者，但有时容易犯迷糊。

ミッシェル (Michelle)：AI 强度动态浮动，思考时间中等。具备局势评估能力，若当前 AI 占据压倒性优势（棋子数量差过大），会主动降低算法强度“放水”，故意选择次优或低收益落子点，以维持游戏平衡感。

5\. 沉浸式交互与演出时序

整个单回合的演出时间线需严格遵循以下顺序：

玩家落子期：玩家点击合法位置，棋子翻转。

玩家台词期：右下角玩家角色的气泡框随机弹出攻击台词（如：“嘿！”、“看招！”），持续 1-2 秒后清空。

电脑拟人思考期：玩家台词结束后，进入 AI 思考延迟阶段（根据角色设定的短/中/长决定延迟秒数）。期间左上角 AI 角色弹出思考台词（如：“哦呀？”、“有点意思呢…”）。

电脑落子期：延迟结束，AI 瞬间完成落子与棋盘翻转，左上角气泡框无缝切换为专属攻击台词，并维持 1-2 秒。随后回合交还给玩家。

6\. 账号与社区模块 (Supabase)

用户系统：基于 Supabase Auth 实现邮箱+密码的注册与登录（暂不开启邮箱强制验证）。在注册流程中，需额外提供“用户名（Username）”的填写环节。该用户名将作为社区互动的唯一公开展示标识，用户的注册邮箱严格保密，不对外公开展示。

数据库 Schema 预设：为了保障数据规范与隐私安全，需要在 Supabase 中创建 profiles（用户信息表）和 comments（评论表）。包含字段如下：

profiles 表（公开的用户配置表）：

id (UUID, Primary Key, Foreign Key 严格关联 auth.users 表的 ID)

username (String, Unique, 用户对外展示的自定义昵称)

created\_at (Timestamp, 默认 now())

comments 表（评论数据表）：

id (UUID, Primary Key)

user\_id (UUID, Foreign Key 关联 profiles.id)

content (Text, 评论正文)

created\_at (Timestamp, 默认 now())

(注意：严禁在此表中存储或向前端传递 user\_email 字段)

权限控制 (RLS 策略)：

读取：所有人（包括未登录用户）均可按时间倒序读取 comments 表，并联合查询（Join）profiles 表以展示评论者的 username。未登录用户评论框显示“请先登录”。

写入：仅限登录用户可发送评论写入 comments 表，且只能以自己的 user\_id 写入。
